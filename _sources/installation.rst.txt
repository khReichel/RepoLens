Deployment and Installation Guide
=================================

This guide provides instructions on how to set up and run the Pulseflow application using Docker.
Pulseflow is composed of three main services:

* a **React.js** frontend
* a Python **FastAPI** backend and
* an **Nginx** gateway.


Prerequisites
-------------

Before starting, ensure you have the following installed on your system:

* **Docker** (version 20.10 or higher)
* **Docker Compose** (version 2.0 or higher)
* **Git** (if installing from source)

Option 1: Installation from Source (GitLab)
-------------------------------------------

Use this method if you want to run the latest development version or modify the source code.

1. **Clone the Repository**

   .. code-block:: bash

      git clone https://gitlab.com/khrl01/pulseflow.git
      cd pulseflow

2. **Prepare Environment**

   Ensure your database files are placed in the ``data/`` directory

   * pulseflow.duckdb - analysis data
   * pulseflow_config.duckdb - configuration data

   You can find empty database files in folder ``examples``.

3. **Import Configuration**

   To populate the configuration database with users, teams, and components, run the database importer.
   This script reads your configuration file and loads the metadata into the configuration database.

   .. code-block:: bash

      python -m src.import_export.importer.main config --config-file /path/to/your/config.yml

4. **Load Analysis Data**

   Next, run the importer again with the ``analysis`` flag to analyze the repository and populate the analysis database.
   This process may take some time depending on the size of the repository.

   .. code-block:: bash

      python -m src.import_export.importer.main analysis --config-file /path/to/your/config.yml


5. **Build and Launch**

   Run the management script to build the images locally and start the services:

   .. code-block:: bash

      chmod +x manage.sh
      ./manage.sh up


   *This will build the Python and Node images, set up the Nginx gateway, and start all containers in detached mode.*

Option 2: Running from Docker Registry
--------------------------------------

Use this method if you want to retrievwe a specific version without handling the source code.

1. **Create a Deployment Folder**

   You only need the ``docker-compose.yml`` and ``nginx.conf`` files.

2. **Login Docker Registry**

    .. code-block:: bash

        docker login registry.gitlab.com -u <token-username> -p <token-passwort>


2. **Pull Images**

   If you have no docker-compose file, you can use this template

    .. code-block:: yaml

        services:
          gateway:
            image: nginx:alpine
            ports:
              - "80:80"
            volumes:
              - ./nginx.conf:/etc/nginx/nginx.conf:ro
            depends_on:
              - backend
              - ui

          backend:
            image: registry.gitlab.com/[dein-benutzer]/[projekt]/backend:latest
            volumes:
              - ./data:/app/dataF
              - ./exports:/app/export
            environment:
              - PYTHONPATH=/app/src
            command: python -m api.run --analysis-db /app/data/pulseflow.duckdb --config-db /app/data/pulseflow_config.duckdb
            healthcheck:
              test: [ "CMD", "curl", "-f", "http://localhost:8000/api/health" ]
              interval: 30s
              timeout: 10s

          ui:
            image: registry.gitlab.com/[dein-benutzer]/[projekt]/ui:latest
            restart: always

   and the configuration of nginx.conf

.. code-block::

        events {
        worker_connections 1024;
        }

        http {
            include       /etc/nginx/mime.types;
            default_type  application/octet-stream;

            server {
                listen 80;

                # 1. Route API requests
                # This matches anything starting with /api/
                location /api/ {
                    # IMPORTANT: If there is a / at the end, “/api/” will be removed.
                    # So “/api/v1/analyze” becomes “http://backend:8000/v1/analyze”.
                    proxy_pass http://backend:8000/;

                    proxy_set_header Host $host;
                    proxy_set_header X-Real-IP $remote_addr;
                    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
                }

                # 2. Route UI requests
                # Everything else goes to the UI container
                location / {
                    proxy_pass http://ui:80;
                    proxy_set_header Host $host;
                    proxy_cache_bypass $http_upgrade;
                }
            }
        }



   Update your ``docker-compose.yml`` to point to the registry images (e.g., ``registry.gitlab.com/khrl01/pulseflow/backend:latest``), then run:

   .. code-block:: bash

      docker compose pull
      docker compose up -d

Accessing the Application
-------------------------

Once the containers are running, the application is accessible via your web browser:

* **Frontend UI:** ``http://localhost`` (Port 80)
* **API Documentation:** ``http://localhost/api/docs`` (Swagger UI)
* **Health Check:** ``http://localhost/api/health``

Maintenance & Database Updates
------------------------------

To update the analysis database safely without data corruption:

1. Place the new ``.duckdb`` file in the ``data/`` directory.
2. Run the replacement command:

   .. code-block:: bash

      ./manage.sh replace /path/to/new_file.duckdb

Troubleshooting
---------------

To inspect the system logs if a service fails to start:

.. code-block:: bash

   docker compose logs -f

* **404 Errors:** Verify the routing in ``nginx.conf``.
* **503 Errors:** The backend is starting up or the database is locked. Check ``docker compose ps`` for health status.
